Make Codex read your design system first.

A tiny Codex skill that tells the agent to load DESIGN.md before building, reviewing, or refactoring UI.

Stop repeating the same spacing, color, component, and "make it less generic" feedback every time you ask for a screen.

Install the skill View DESIGN.md example

Codex can build fast. This makes it build in your product's language.

Without a design source of truth, every new screen can quietly invent a new one. This skill gives Codex a repeatable first step: read the repo's design rules, then make UI decisions.

Without this skill

  • New accent colors show up uninvited.
  • Cards get softer, bigger, or shinier screen by screen.
  • Spacing and type decisions depend on the last prompt.
  • Reviews become a loop of "closer, but less generic."

With this skill

  • Codex reads DESIGN.md before UI work.
  • Tokens and component rules stay in the repo.
  • Visual decisions have a source of truth.
  • Rendered UI gets checked at desktop and mobile sizes.

One small skill. One repeatable design habit.

It does not replace taste. It gives Codex a place to find it.

01 Find the design file

Prefer a root DESIGN.md in the target repo; use this kit's bundled example when you need a starter.

02 Apply the rules

Use documented colors, type, spacing, radii, motion, components, and no-go zones.

03 Protect consistency

Avoid inventing new visual systems when the repo already has a perfectly good one.

04 Verify the screen

When frontend code changes, check the rendered UI at desktop and mobile widths.
